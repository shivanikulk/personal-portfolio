Building a Portfolio Landing Page

This project explored how to use AI as a serious design and implementation partner rather than a shortcut generator. The goal was to ship a portfolio landing page that felt configurable, production-minded, and visually deliberate, while documenting a workflow that combined rapid iteration with product-level judgment.

Why this matters The artifact is not just the page itself. The real deliverable is the workflow: defining structure, pressure-testing UI decisions, and translating design intent into an editable system that can evolve without rewriting code.

Project framing

01

The starting point was a lightweight portfolio site, but the design challenge was broader than building a landing page. The objective was to create a portfolio experience that could scale across different project types, support varied media formats, and feel intentional.

How Codex and Claude were used

02

The workflow deliberately split responsibilities between models. Codex was used as the implementation partner: it inspected the codebase, updated the rendering logic, created configuration hooks, and translated design requirements into working HTML, CSS, and JavaScript. Claude was used as a critique and synthesis partner: helping refine narrative clarity, stress-test framing, and push the articulation of design intent beyond a purely functional explanation.

The key pattern was not "ask AI for a page." It was: define an intent, evaluate the output critically, tighten the system, and use each model for the kind of reasoning it is strongest at.
  • Codex accelerated implementation, debugging, and component-level iteration.
  • Claude helped sharpen rationale, communication, and editorial quality.
  • The final direction still depended on human judgment for priorities, taste, and tradeoff decisions.

Iterative design process

03

The page evolved through many small decisions rather than one large redesign. Early iterations focused on structure: separating profile content from project content, moving portfolio data into JSON, and making projects easy to add or reorder. Once the content model stabilized, the next iteration focused on layout behavior and card hierarchy.

Step 1: Systemize content

Move profile, links, and projects into structured JSON so the experience could be edited without touching rendering code.

Step 2: Improve card semantics

Differentiate project metadata, description, and caption to create stronger scanning behavior on the tile itself.

Step 3: Add layout controls

Introduce item-level sizing so projects could behave like editorial modules instead of identical repeated thumbnails.

Step 4: Fix responsive behavior

Replace brittle fixed-height logic with more resilient responsive sizing so images and GIFs remained readable across breakpoints.

Design decisions

04

The strongest design decision in this project was to treat the project tiles as narrative surfaces instead of decorative containers. Descriptions were moved into the tile, captions were separated from the supporting story, and the card sizing model was made explicit with small, medium, and large states. This created clearer hierarchy and gave the layout a more editorial, portfolio-specific rhythm.

  • Sections became optional so the portfolio could support both grouped case studies and simple project streams.
  • Item-level size controls made the layout more expressive without introducing a heavy CMS.
  • Text rendering was updated to support line breaks and tabs directly from JSON, improving control over copy presentation.
  • Image and GIF handling was tuned to preserve legibility rather than aggressively crop media for fill.

Key outcomes

  • Shipped a JSON-driven portfolio structure that is editable without touching layout code.
  • Created a more flexible project-card model with small, medium, and large behaviors.
  • Improved responsive handling for mixed media, including animated GIFs.
  • Added project-specific routes so exploratory work can be documented as deeper case studies.

Working principle

AI was most useful when treated as a collaborative layer in the design process, not as a replacement for product judgment.

Scope of contribution

  • Defined the narrative structure of the case study.
  • Directed the visual and interaction refinements across iterations.
  • Used AI models differently based on their strengths: implementation, critique, and synthesis.
